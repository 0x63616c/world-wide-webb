Routes and TLS

Apps declare exposure. Platform owns the network machinery.

Networking is load-bearing. The platform must preserve TLS, Cloudflare, cert-manager, UniFi split DNS, firewall rules, and captive portal behavior while moving to product hostnames.

There is no generic LAN web class in the North Star. Captive Portal is the only LAN-like web surface today, and it has special captive-network semantics.

M1 Foundation Contract

  • @repo/platform implements exposure intent declarations named privateWeb(), captivePortalWeb(), and internalService().
  • These declarations derive single-label hostnames from product DNS codes and the homelab target domain, for example app--cc.worldwidewebb.co and app--cp.worldwidewebb.co.
  • M1 does not change live Cloudflare routes, Access policies, cert-manager certificates, UniFi DNS, guest portal redirects, LAN forwarding, or iOS URLs.
  • captivePortalWeb() carries an explicit human-review marker because later implementation touches UniFi, LAN forwarding, DNS, and TLS together.

Hostname Rules

  • Product namespace hosts are not created by default: no bare cc.worldwidewebb.co or cp.worldwidewebb.co.
  • Service hostnames are explicit and flattened to a single label, for example app--cc.worldwidewebb.co.
  • Short DNS labels are derived by the networking primitive from the product slug: control-center to cc and captive-portal to cp. The service host is then <host>--<code> (e.g. app--cc), one label, joined by a double hyphen.
  • Single-label hosts are covered by the free root wildcard: *.worldwidewebb.co covers both dashboard.worldwidewebb.co and app--cc.worldwidewebb.co. This is the whole reason hosts are flattened (app--cc) instead of nested (app.cc): a 2-label host would need paid Cloudflare ACM, a single label does not (www-kbiy).
  • APIs do not automatically get external hostnames. External API routes are app-specific and declared only when needed.
  • Public routes are opt-in. New routes default private or internal by platform policy.

TLS Contract For Product Hosts

Product hosts are flattened to a single label (app--cc), so the free Cloudflare Universal SSL *.worldwidewebb.co covers every Cloudflare-proxied route automatically. No ACM, no per-host edge cert. The platform still makes certificate coverage explicit for every exposure (the exact-host coverage model in @repo/platform).

Current State To Respect

Cloudflare Tunnel

Dashboard, Storybook, Drizzle, hooks, and Portainer have tunnel/DNS state today. Captive Portal is intentionally absent from Cloudflare tunnel routes.

Cloudflare Access

Storybook and Drizzle are deployed with Access today. Dashboard is not Access-gated yet, even though the North Star private model requires it.

Captive Portal TLS

Captive Portal uses cert-manager with Cloudflare DNS-01 to issue a real certificate while traffic stays LAN-only.

UniFi Path

Captive Portal depends on UniFi guest network config, split-horizon DNS, firewall/walled-garden behavior, and LAN forwarding into k8s.

Captive Portal Network Contract

Captive Portal is a platform networking primitive because it spans app code, k8s, host networking, UniFi, DNS, certificates, and device captive network behavior.

  • Configure UniFi local DNS for app--cp.worldwidewebb.co to the Mini LAN path.
  • Update any host-level forwarding that still points at the old control-center/captive-portal service identity.
  • Configure guest portal redirect URL to the new app host.
  • Preserve or replace firewall/walled-garden rules deliberately.
  • Keep DNS-01 certificate issuance working for the LAN-only hostname.
  • Verify from the real guest network and from iOS captive network assistant, not just desktop browser.
  • Do not route Captive Portal through normal public Cloudflare unless explicitly redesigned.

TLS Cutover Rules

  • Do not break TLS during hostname migration.
  • For nested product hosts, create exact-host or product-wildcard certificate coverage before traffic moves.
  • Inventory every old hostname before changing route names.
  • Update cert-manager DNS names and mounted TLS secrets before switching traffic.
  • Update iOS app URLs before retiring old kiosk hostnames.
  • Verify Cloudflare routes, Access policies, cert-manager certificate readiness, UniFi DNS, and captive portal LAN path independently.
  • Hard cutover is allowed, but must have a checklist and rollback path.

Cloud Target Constraint

The platform should be target-aware, but this migration ships homelab k3s only. Homelab-specific dependencies such as Home Assistant, UniFi, NAS backups, and captive portal LAN routing must live behind target/platform primitives rather than leaking into every product.

Networking mistakes take products offline. Treat TLS, Access, UniFi, and captive portal behavior as acceptance criteria, not implementation details.
