End state

A really good codebase to work in.

The repo is an internal product platform. Adding a new app feels boring: define the app, add services, ask for a database if needed, declare required secrets, and get local Tilt, CI, deploy, routing, TLS, backups, and docs by default.

North Star Statement

world-wide-webb is a product platform repo. The platform team owns shared primitives. App teams consume simple declarations. The platform makes the right thing easy and the wrong thing hard.

The goal is not just to move text-your-ex into this repo. The goal is to reshape the repo so four products can run independently on one homelab k3s platform today, while leaving room for future runtime targets.

What Great Feels Like

Obvious Boundaries

products/control-center, products/captive-portal, products/text-your-ex, and products/amp make ownership visible. Shared code lives in platform packages, not hidden inside a product.

Isolated Products

Each product gets its own namespace, deploy workflow, route declarations, database cluster, backup path, runtime config, and test/verification story.

Opinionated Platform

The platform owns naming, labels, TLS, secrets, backup jobs, restore conventions, route mechanics, and Kubernetes object wiring.

No String Soup

Product identity is defined once. Namespaces, hostnames, backup paths, image names, secret names, labels, and Pulumi resource names derive from typed app definitions.

Local First

Every product is runnable locally through Tilt. iOS products expose the same manual actions: sync, open Xcode, and run simulator.

Production Safe

Every milestone has tests, verification, rollback thinking, and data safety. Internal refactor-only tasks are allowed only when they create a safe seam for production work.

Platform Principles

  • Act like an infrastructure platform team.
  • App teams consume simple primitives.
  • Make the right thing easy.
  • Standardize shared infra behavior aggressively.
  • Expose only necessary customization.
  • Do not expose knobs just because the underlying tool has knobs.
  • Backups are a platform guarantee, not an app option.
  • Secrets are centrally managed, but product and service access is explicitly declared.
  • Shared secrets are allowed when they are genuinely shared account-level or platform-level secrets.
  • Avoid magic strings by deriving names from typed app context.
  • Do not duplicate product infra wiring.
  • Do not over-abstract before real products prove the shape.

App Team Experience

The app team API should feel like this:

const app = defineApp({
  id: "cp",
  slug: "captive-portal",
});

app.service("web", {
  image: images.product("web"),
  exposure: captivePortalWeb(),
});

app.service("api", {
  image: images.product("api"),
  exposure: internalService(),
  secrets: [secrets.unifi.localApiKey, secrets.resend.apiKey],
});

app.database({
  size: "5Gi",
});

The app team declares intent. The platform derives namespace, DNS, TLS, route, service, database, secret, backup, and deploy details.

Explicit Scope Boundaries

In this migration: all four products ship to homelab k3s, the platform primitives exist, per-product CNPG Postgres clusters exist, backups are automatic, and CI/deploy is product-aware.

AMP: build and deploy it in parallel as a minimal production management-plane shell. It proves the platform path for a new internal app, but it should not block the three existing product migrations from progressing.

Cloud: the platform design is target-aware, but no cloud k3s cluster or cloud storage migration is required in this migration.

SQLite/libSQL: not the end state. The database standard is Postgres through CNPG.

Generic LAN web: not a platform exposure class until a real product needs it. Captive Portal is a special captive-network exposure, not a generic LAN app.

North Star docs describe the destination. Phasing and temporary compromises belong in the migration plan.
