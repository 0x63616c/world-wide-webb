Current state

What we learned before planning the migration.

These notes preserve the investigation results that informed the North Star. They are not all end-state requirements, but they explain why the platform shape exists.

Current Cluster

  • Platform namespaces exist for cert-manager, external-secrets, cnpg-system, and kube-system.
  • One product-ish namespace exists today: control-center.
  • control-center currently contains API, web, worker, captive portal, cloudflared, CNPG Postgres, and cronjobs.
  • Captive Portal is not isolated today.

Captive Portal Coupling

Captive Portal currently lives across multiple Control Center areas.

This is the strongest evidence that Captive Portal needs a real product extraction, not just a folder move.

Text Your Ex Shape

  • Current repo is self-contained with web, server, e2e, ios, fastlane, and a Dockerfile.
  • Runtime is React/Vite plus Bun/Hono API.
  • Current data is SQLite through bun:sqlite.
  • Server boot currently calls ensureSeed(), which would seed demo data into an empty production database unless changed.
  • Capacitor app is bundled mode with app id co.worldwidewebb.textyourex.
  • TestFlight workflow already exists and supports VITE_API_BASE.
  • Migration-friendly, but needs Postgres, seed gating, platform CI/deploy, and product routes.

Postgres Versus SQLite

The planning conversation considered SQLite, Litestream, LiteFS, and libSQL/sqld. The end-state decision is Postgres.

OptionLearningDecision
Plain SQLiteSimple and fine for one API pod, but app owns the file and multi-writer behavior is constrained.Not the platform standard.
LitestreamBackup/replication helper for SQLite, not server mode.Not needed.
LiteFSPowerful but operationally specific and less boring outside environments designed around it.Rejected for this migration.
libSQL/sqldServer-mode SQLite-like option, but less standard than Postgres and requires validation.Rejected as end state.
Postgres/CNPGAlready running, operationally known, supports current query patterns and queue semantics.Chosen platform standard.

Postgres Compatibility Findings

  • Current schema uses Drizzle Postgres primitives: pgTable, jsonb, serial, timestamp with time zone, and defaultNow().
  • Migrations are Postgres SQL.
  • The durable job queue depends on FOR UPDATE SKIP LOCKED, row locks, make_interval, now(), raw SQL, and transactions.
  • Control Center would require a real rewrite to move away from Postgres.
  • Captive Portal is easier to move because it already has a repository abstraction, but the end-state database standard is still Postgres.

Current Infra Hardcoding

  • Namespace hardcoded to control-center.
  • Images use ghcr.io/0x63616c/control-center-<service>.
  • Secrets use legacy Control-Center-prefixed service names.
  • Pulumi image digest config uses a legacy Control-Center-specific namespace.
  • Database service hardcoded as control-center-rw.
  • Dashboard hostname is old dashboard.worldwidewebb.co.
  • Captive Portal hostname is old captive-portal.worldwidewebb.co.
  • iOS kiosk config points at the old dashboard URL.
  • Cloudflare route/adopt-only state still has legacy comments and hosts.

CI/Deploy Findings

  • One CI workflow currently controls tests, image builds, and deploy.
  • Path filters separate image rebuilds somewhat, but deploy is one stack-wide Pulumi run.
  • Shared changes rebuild broad sets of images.
  • Product isolation matters more than perfect CI speed in this migration.
  • CI optimization can be partial. Correctness, traceability, and blast-radius reduction come first.

Cloud Target Decision

The conversation considered cloud runtime and stretched home/cloud k3s. The decision for this migration is homelab-only implementation with target-aware design.

  • Do not create a stretched k3s cluster across homelab and cloud.
  • Do not ship a cloud target in this migration.
  • Do design platform primitives so future targets are possible.
  • Keep homelab-specific dependencies behind target/platform abstractions.
