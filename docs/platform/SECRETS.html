Centralized but explicit

Secrets stay central. Access becomes declarative.

1Password Homelab remains the source of truth. Some secrets are intentionally shared. The important change is that every product and service declares which secrets it needs, instead of inheriting a Control Center shaped bag of env vars.

Principles

  • All real secrets live in 1Password Homelab.
  • External Secrets Operator remains the cluster sync path.
  • Secret management can remain centralized.
  • Shared account-level secrets are allowed when they are truly shared.
  • Product and service access to a secret must be explicitly declared.
  • Raw "Item/field" strings should not be scattered across product infra.
  • Platform derives Kubernetes secret names and mount paths from app/service context.
  • Product services mount only the secrets they declare.

Target Shape

Prefer a typed or declarable secret catalog over string refs in product code.

const secrets = defineSecretCatalog({
  unifi: {
    localApiKey: op.item("UniFi").field("local_api_key"),
  },
  resend: {
    apiKey: op.item("Resend").field("credential"),
    fromAddress: op.item("Resend").field("from-address"),
  },
  github: {
    ghcrPat: op.item("GitHub Personal Access Token").field("token"),
  },
  cloudflare: {
    tunnelToken: op.item("Cloudflare Tunnel").field("connector_token"),
  },
});

captivePortal.service("api", {
  secrets: [
    secrets.unifi.localApiKey,
    secrets.resend.apiKey,
    secrets.resend.fromAddress,
  ],
});

Shared Versus Product Secrets

KindExamplesRule
Platform sharedGHCR pull token, Pulumi token, Tailscale OAuth, Apple signing credentialsManaged centrally. Products should not duplicate or rename just for isolation theater.
Integration sharedUniFi API key, Cloudflare API, Resend account credentialCan be shared, but each service that receives it must explicitly declare use.
Product-ownedProduct database password, product API token, app-specific webhook secretLives under product identity and syncs only to that product namespace.
Build/releaseVITE_API_BASE, bundle id, iOS scheme, TestFlight settingsCan be product-specific GitHub variables/secrets or generated from product metadata where possible.

Current Problem To Fix

The current secrets map is service-oriented and Control-Center-specific. It syncs secrets into legacy Control-Center-prefixed service secrets, and Captive Portal API secrets are mixed into the shared Control Center API service.

// Current shape to retire:
SERVICE_SECRETS = {
  api: {
    UNIFI_API_KEY: "UniFi/local_api_key",
    RESEND_API_KEY: "Resend/credential",
    POSTGRES_PASSWORD: "Control Center Postgres/password",
  },
};

Target shape is product/service scoped and derived from the app context.

External Secrets Contract

  • External Secrets Operator remains installed as platform infrastructure.
  • The 1Password ClusterSecretStore can remain central.
  • Each product namespace gets only the ExternalSecrets it needs.
  • Secret target names are derived, for example <product>-secrets-<service> or another platform-owned convention.
  • Database auth secrets are generated and managed by the database primitive, not hand-declared per product.
  • Adding a new secret should include a save script when it is a new credential, following the 1Password convention.
Centralized secrets are fine. Undeclared secret access is not.
