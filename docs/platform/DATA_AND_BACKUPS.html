Data guarantee

Databases are a platform feature.

The end state is all Postgres through per-product CNPG clusters. If a product declares a database, it gets durable storage, secrets, service discovery, backups, and restore validation by default.

Database Standard

End state: all products use Postgres through CloudNativePG.

Topology: one CNPG Postgres cluster per product.

Default database shape: one application database and one application user inside that product cluster. Add more databases only when the product has a concrete boundary that needs them.

Not the standard: SQLite, libSQL, sqld, LiteFS, or embedded database files.

Reason: current Control Center already depends on Postgres semantics, CNPG exists, backups exist, and Postgres is the boring production default.

What size Means

database({ size: "5Gi" }) means the Kubernetes PVC storage request for that product's Postgres data directory. It is effectively the disk volume size available to Postgres for table data, indexes, WAL, engine files, and overhead.

  • It is not exactly the logical maximum database size because Postgres overhead and WAL use space too.
  • If the volume fills, Postgres can stop accepting writes or fail.
  • CNPG/local-path can resize volumes when configured correctly.
  • Size is a valid app-level knob because product data volume differs.

Backup Standard

Backups are a platform guarantee. App code does not decide whether backups exist, where they live, or how they are named. The platform derives that from app identity.

  • Every product database gets a nightly backup CronJob.
  • Backups write to the NAS under one platform-owned root.
  • Backups are retained forever.
  • Restore validation is required before data migration work is considered complete.
  • Future retention policies must be platform-level decisions, not per-product ad hoc scripts.

Data Cutover Contract

  1. Inventory: identify source tables, owners, write paths, read paths, cronjobs, and workers.
  2. Target schema: preserve table names/types where product behavior depends on them. Keep IDs stable.
  3. Export: take both a custom-format dump and a plain SQL gzip dump before cutover.
  4. Restore rehearsal: restore into a scratch database, run migrations, validate row counts and semantic queries.
  5. Write freeze: pause writers for a short maintenance window when needed.
  6. Final snapshot: export again after freeze, restore to target, validate counts and key queries.
  7. Cutover: deploy app pointing at target database and verify health, flows, and backups.
  8. Rollback: keep old database and dumps untouched until the new database survives backup and restore validation.
  9. Post-cutover: run a backup, restore it into scratch, and compare counts again.

Per-Product Data Notes

Control Center

Current data stays safe throughout the migration, then cuts over to the Control Center product CNPG cluster with rehearsal and validation. A short maintenance window is acceptable. The job queue depends on Postgres row locks and stays Postgres.

Captive Portal

Portal tables move into Captive Portal's product database: guests, codes, attempts, and authorizations. Preserve enough history and authorization state to avoid breaking active guest access.

Text Your Ex

Current prototype uses SQLite. Production standard is Postgres. Prod starts empty, while local demo seed remains explicit and non-prod.

AMP

AMP declares a database only when it needs durable product state. If it declares one, it gets the same CNPG and backup guarantees.

A backup that has never been restored is not proven. Restore validation is part of the platform contract.
