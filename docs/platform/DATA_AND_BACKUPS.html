Data guarantee

Databases are a platform feature.

The end state is all Postgres through per-product CNPG clusters. If a product declares a database, it gets durable storage, secrets, service discovery, backups, and restore validation by default.

Database Standard

End state: all products use Postgres through CloudNativePG.

Topology: one CNPG Postgres cluster per product.

Default database shape: one application database and one application user inside that product cluster. Add more databases only when the product has a concrete boundary that needs them.

Not the standard: SQLite, libSQL, sqld, LiteFS, or embedded database files.

Reason: current Control Center already depends on Postgres semantics, CNPG exists, backups exist, and Postgres is the boring production default.

What size Means

database({ size: "5Gi" }) means the Kubernetes PVC storage request for that product's Postgres data directory. It is effectively the disk volume size available to Postgres for table data, indexes, WAL, engine files, and overhead.

  • It is not exactly the logical maximum database size because Postgres overhead and WAL use space too.
  • If the volume fills, Postgres can stop accepting writes or fail.
  • CNPG/local-path can resize volumes when configured correctly.
  • Size is a valid app-level knob because product data volume differs.

Backup Standard

Backups are a platform guarantee. App code does not decide whether backups exist, where they live, or how they are named. The platform derives that from app identity.

  • Every product database gets a nightly backup CronJob.
  • Backups write to the NAS under one platform-owned root.
  • Backups are retained forever.
  • Restore validation is required before data migration work is considered complete.
  • Future retention policies must be platform-level decisions, not per-product ad hoc scripts.

M1 Foundation Contract

  • defineProductDatabase() models the per-product CNPG contract in @repo/platform, including database name, owner, auth secret, read-write service, storage, and resource defaults.
  • defineDatabaseBackup() makes a backup declaration mandatory for every product database. There is intentionally no per-app disable knob.
  • The Control Center manifest uses compatibility options to represent the current control-center cluster, control_center database, cc-postgres-auth secret, and pg-backup CronJob.
  • The Captive Portal manifest represents the product-owned captive-portal CNPG cluster, captive_portal database, captive-portal-rw service, captive-portal-postgres-auth secret, and captive-portal-pg-backup CronJob.
  • The future NAS path derives as backups/world-wide-webb/<product>/postgres, but M1 does not move the live backup destination from the current backups/postgres path.

M2 Backup Adapter Contract

  • infra/src/crons.ts adapts the platform DatabaseBackup declaration into the concrete CronJobSpec through postgresBackupCronSpec().
  • The adapter derives CronJob name, schedule, PG image, database host/name, auth secret mount, backup mount, NFS export, NAS subpath, and dated gzip filename from the platform declaration instead of Control Center-only constants.
  • The default product path remains backups/world-wide-webb/<product>/postgres, matching the platform NAS root.
  • Control Center intentionally uses a compatibility declaration for now: CronJob pg-backup, schedule 0 1 * * *, database control_center, service control-center-rw, auth secret cc-postgres-auth, and NAS subpath backups/postgres.
  • Captive Portal uses the standard product path now: CronJob captive-portal-pg-backup, schedule 15 1 * * *, database captive_portal, service captive-portal-rw, auth secret captive-portal-postgres-auth, and NAS subpath backups/world-wide-webb/captive-portal/postgres.
  • Moving Control Center's live backup destination to backups/world-wide-webb/control-center/postgres is a future reviewed production-path migration, not an incidental primitive refactor.

M5 Captive Portal Database Contract

  • Captive Portal has a product-owned CNPG cluster named captive-portal with app database captive_portal, owner postgres, read-write service captive-portal-rw, and local-path storage.
  • The app credential lives in the CNPG basic-auth Secret captive-portal-postgres-auth, bridged from op://Homelab/Captive Portal Postgres/password. It does not reuse the Control Center Postgres credential.
  • The Captive Portal API declares only its required secret inputs: POSTGRES_PASSWORD, RESEND_API_KEY, RESEND_FROM, UNIFI_API_KEY, WIFI_PASSWORD, and WIFI_SSID.
  • The nightly backup writes captive_portal-YYYYMMDD.sql.gz to /volume1/Homelab/backups/world-wide-webb/captive-portal/postgres/.
  • This provisions the destination database and backup path only. Moving portal tables out of Control Center still requires reviewed export, scratch restore, final snapshot, validation, and cutover.

Data Cutover Contract

  1. Inventory: identify source tables, owners, write paths, read paths, cronjobs, and workers.
  2. Target schema: preserve table names/types where product behavior depends on them. Keep IDs stable.
  3. Export: take both a custom-format dump and a plain SQL gzip dump before cutover.
  4. Restore rehearsal: restore into a scratch database, run migrations, validate row counts and semantic queries.
  5. Human review: stop before any production database apply, final snapshot, or non-scratch restore. Record the approved source, target, scratch cluster, and rollback plan.
  6. Write freeze: pause writers for a short maintenance window when needed.
  7. Final snapshot: export again after freeze, restore to target, validate counts and key queries.
  8. Cutover: deploy app pointing at target database and verify health, flows, and backups.
  9. Rollback: keep old database and dumps untouched until the new database survives backup and restore validation. For Captive Portal, leave the old Control Center portal tables untouched and scale any new product DB consumers to zero if provisioning fails.
  10. Post-cutover: run a backup, restore it into scratch, and compare counts again.

M2 Snapshot And Restore Tooling

scripts/pg-snapshot-restore.sh is the reusable platform-era proof tool for product database snapshots and scratch restores. It is additive and does not replace the live backup CronJob.

  • It captures exact row counts for every non-system schema/table, including drizzle.__drizzle_migrations.
  • It writes both a custom-format dump and a plain SQL gzip dump.
  • It restores only into an explicit scratch target; production and control-center are rejected as scratch targets.
  • It captures scratch row counts and fails non-zero on any mismatch with a stop message.
  • It never reads or prints password values; auth stays inside CNPG pods.
scripts/pg-snapshot-restore.sh --dry-run --source production --scratch <scratch-cluster>
scripts/pg-snapshot-restore.sh --source production --scratch <scratch-cluster> --output-dir ./.pg-snapshots
scripts/pg-snapshot-restore.sh --compare-counts source-counts.tsv scratch-counts.tsv

For Captive Portal, use the same commands against an approved portal source and scratch target before the product database becomes authoritative. The restore proof must compare row counts before any production apply or cutover.

Human Review Checkpoints

  • Production dump capture requires explicit human approval before any dump command runs.
  • Final pre-cutover snapshots require explicit approval and a documented writer-quiesce plan.
  • Any non-scratch restore, including restore into a product production database, requires explicit approval.
  • Deleting old dumps, deleting scratch artifacts, changing backup paths, or removing rollback databases requires explicit approval.

Per-Product Data Notes

Control Center

Current data stays safe throughout the migration, then cuts over to the Control Center product CNPG cluster with rehearsal and validation. A short maintenance window is acceptable. The job queue depends on Postgres row locks and stays Postgres.

Captive Portal

Portal tables move into Captive Portal's product database: guests, codes, attempts, and authorizations. Preserve enough history and authorization state to avoid breaking active guest access.

Text Your Ex

Current prototype uses SQLite. Production standard is Postgres. Prod starts empty, while local demo seed remains explicit and non-prod.

AMP

AMP declares a database only when it needs durable product state. If it declares one, it gets the same CNPG and backup guarantees.

A backup that has never been restored is not proven. Restore validation is part of the platform contract.
