# P1.6 — Relocate photo-path-migration (unweld photo tiles) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move `photo-path-migration` out of `apps/api/src/services/` (where it sits beside the two foldable photo services) into a new `apps/api/src/startup/` module, so it reads as an apps/api shell startup orchestrator — not a service — and booth/wake can fold into independent features later without one tile's code importing the other.

**Architecture:** This is a pure file relocation plus import repoints — no behavior change. The module orchestrates BOTH the booth and the wake photo stores at api boot. It depends on the apps/api-owned db schema (`boothPhoto`, `wakePhoto` tables) and on `defaultBoothPhotoRoot()` / `defaultWakePhotoRoot()` from the two services, so it CANNOT hoist to `packages/core` or `packages/platform` (packages may not import from `apps/api`). The only correct home is the apps/api shell, where importing both feature services is the allowed dependency direction (apps/api → features). Relocating it from `services/` to a dedicated `startup/` folder makes that role explicit and separates it from the two services that WILL fold.

**Tech Stack:** TypeScript, Bun, Vitest, drizzle-orm, `@www/core`, `@www/logger`.

## Global Constraints

- Dependency direction: `apps/api → features` is allowed; `features → apps/api` and `packages → apps/api` are Biome-banned. This module stays in `apps/api` precisely because it must reach both photo services.
- No shim. Only 2 importers, and it is not an env-bound singleton — a shim would be dead code that knip's zero-tolerance pass fails on. (Mirror P1.4 media-path, which did NOT shim; unlike HA in P1.1 which kept shims for a many-consumer env-bound singleton.)
- Do NOT hand-edit `features/_generated/*`; run `bun run apps:gen`.
- `bun build` reads the CWD tsconfig (not entry) — irrelevant here (no build step touched) but do not `cd` away from repo root during verify unless a command needs it.
- Before push: `git pull --rebase --autostash` (peers push main constantly).
- Stage EXPLICIT paths, NEVER `git add -A`. Verify with `git show --stat HEAD` that only the four intended files are in the commit.
- Commit message has NO backticks.
- Do NOT commit until the FULL verify chain passes with real output evidence.

## Decision Record

- **Destination:** `apps/api/src/startup/photo-path-migration.ts` (+ `apps/api/src/startup/photo-path-migration.test.ts`). New `startup/` directory under `apps/api/src/`.
- **Why not `packages/platform`/`packages/core`:** the module imports `../db/schema` (`boothPhoto`, `wakePhoto`) and the two service root helpers. Those are apps/api-owned; packages cannot import them. The master-plan-listed "OR packages/platform" option is ruled out by real dependencies. The startup-module option is the correct one.
- **Shim vs repoint:** REPOINT. 2 importers only (`server.ts` + the module's own test), not an env-bound singleton with many consumers. A re-export shim at the old path would be an unused export → knip failure.
