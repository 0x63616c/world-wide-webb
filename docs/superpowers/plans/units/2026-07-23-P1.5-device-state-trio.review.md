## 1. [MAJOR] climate-enforcer symbol enumeration is incomplete (4 symbols dropped)

Both the "Current state" analysis (lines 76-77) and Phase 2c (lines 259-261)
enumerate the climate-enforcer import from `./device-state-mapping` as only five
symbols: `isClimateState, MappedHaState, mapHaToReported, rememberedClimateDesired,
sanitizeClimateDesired`.

The real import block (`climate-enforcer-service.ts:35-45`) pulls **nine**:

```
climateSetpointsObservable, climateStateConverged, DeviceKind,
isActivelyConditioning, isClimateState, MappedHaState, mapHaToReported,
rememberedClimateDesired, sanitizeClimateDesired
```

Missing from the plan: `climateSetpointsObservable`, `climateStateConverged`,
`DeviceKind` (all concern-1, in `@www/core` via `merge`/`schema`) and
`isActivelyConditioning` (concern-2, in the new `ha-mapping.ts` — it IS in the
plan's ha-mapping export list at line 205, and `climateSetpointsObservable`/
`climateStateConverged` are consumed at `:100-112`, verified).

Fix: all nine resolve from `@www/core` after the split, so a **wholesale swap of
the specifier** `from "./device-state-mapping"` → `from "@www/core"` (keeping the
symbol list intact) is correct and complete. Update the plan's two lists to name
all nine, and add an explicit instruction: "repoint by changing the specifier,
do NOT prune symbols." Risk if left: an implementer editing symbol-by-symbol per
the plan's list silently drops four live imports and reds typecheck.

## 2. [MINOR] Phase 3d PLACEHOLDER "grep found none" is factually wrong

Line 550-552 asserts no enforcer/weather test imports the `integrationSyncStatus`
symbol directly ("grep found none"). It does:
`light-enforcer-service.test.ts:38` — `import { integrationSyncStatus, lampMode }
from "../db/schema";` — and keys its `mockDbSelect` table-chain by that object at
`:291` (`[integrationSyncStatus, []]`).

This still works with zero code change **because** the plan re-exports the table
from `db/schema.ts` via `export { integrationSyncStatus } from "@www/core"`, an
identifier re-export that preserves object identity — so the test's imported
table === the core table the pg store selects from, and the mock key matches.

Fix: correct the claim, and state the load-bearing reason explicitly in the
gotchas: "the 4 db-mock tests survive only because the re-exported table is the
SAME object the core pg store queries; do not replace the re-export with a fresh
local `pgTable` def." That invariant is currently implicit.

## 3. [MINOR] `updatedAtUtc` not bumped on conflict-update — preserve, don't "fix"

The original `heartbeat` write (`integration-heartbeat.ts:37-42`) omits
`updatedAtUtc` from the `onConflictDoUpdate` `set`, so on every update the column
keeps its first-insert value (only `defaultNow()` at insert). The plan's `pg.ts`
faithfully mirrors this (no `updatedAtUtc` in `set`) — correct for "no behavior
change". The `memory.ts` adapter DOES set `updatedAtUtc: now` on every write; a
harmless divergence since memory is test-only and no assertion reads it. Note it
in the plan so it isn't mistaken for a bug and "harmonized" the wrong way.
