## Findings

### [BLOCKER] 1 — §4 mis-states the `__stories__/factory` shared-helper imports; 5 stories files will ship broken relative imports

The `__stories__/factory` helper (`apps/web/src/components/tiles/__stories__/factory.ts`,
exports `defineTileMeta` + `modalDocsParameters`) is NOT in the move set — it stays
in apps/web. Five of the moving stories files reference it **relatively**, and after
`git mv` those relative paths no longer resolve:

- `ClimateTileView.stories.tsx:8` → `import { defineTileMeta } from "./__stories__/factory";`
- `ClimateModalComfortPresetsFan.stories.tsx:11` → `import { modalDocsParameters } from "../__stories__/factory";`
- `ClimateModalHouseThermalMap.stories.tsx:9` → same `../__stories__/factory`
- `ClimateModalMultiZoneGrid.stories.tsx:13` → same
- `ClimateModalScheduleTimeline.stories.tsx:14` → same

The plan §4 gets this wrong two ways: (a) it lists the tile-story factory import
under **"Keep as `@/…`"**, but the real import is relative `./__stories__/factory`,
so "keep" leaves a dangling path — it must be **rewritten** to
`@/components/tiles/__stories__/factory`; (b) it never mentions `modalDocsParameters`
or the four modal-story `../__stories__/factory` imports at all.

**Fix:** in all five moved stories files rewrite the factory import to the absolute
form `@/components/tiles/__stories__/factory` — exactly what the proven weather fold
did (`features/weather/web/WeatherNowView.stories.tsx:8` and
`features/weather/web/views/WeatherModalWeekOutlook.stories.tsx:7` both use the `@/`
form). Add this to §4's "Rewrite → `@/`" list; remove factory from the "Keep as `@/`"
list. (typecheck + the `*.stories.test.tsx` collection catch this on verify, so it is
recoverable fix-forward — but the plan's written instruction actively mis-directs, so
correct it before implementation.)

### [MINOR] 2 — `CLIMATE_DEVICE_ID` reader count is off by one

§5.1 says the const is read by "service, enforcer, controls-service, plus two tests."
Real importers: `climate-service.ts`, `climate-enforcer-service.ts`,
`controls-service.ts`, and only ONE test (`climate-enforcer-service.test.ts`). The
service test (`climate.test.ts`) does not import it. Harmless — the re-export
mechanism keeps every staying importer zero-churn regardless of count — but correct
the tally to avoid a false "did I miss a test" hunt during verify.

### [MINOR] 3 — `packages/core` index re-export line not spelled out; `FEATURE_MANIFESTS` "alphabetical" is cosmetic

(a) §5.1/§7.6 says add `CLIMATE_DEVICE_ID` to a new `device-state/identity.ts`
"re-exported from index.ts" but doesn't state the exact line. The index uses
per-file `export * from "./device-state/…"`, so the implementer must add
`export * from "./device-state/identity";`. State it explicitly. (b) §7.3 says add
`acManifest` "alphabetical with the other feature-manifest imports", but the real
`FEATURE_MANIFESTS` array is insertion-ordered, not alphabetical
(guestWifi/network/tesla/dogcam/weight/deploys/notif/weather). Array position is
functionally irrelevant (dedup + validate handle placement), so this is cosmetic —
just don't let it read as a required ordering.

### Note (not a finding) — §5.4 test mock placeholder is correctly deferred

The current `climate.test.ts` mocks `../integrations/homeassistant` stubbing only
`ha` (isConfigured/getEntities/callService); it never mocks `deviceStateStore` and
injects `createInMemoryDeviceStateStore` into each call. Retargeting to
`vi.mock("./deps", …)` with only `ha` is safe **iff** no tested path falls back to
the default store — which is exactly what the §5.4 `PLACEHOLDER (implementer)` tells
the implementer to confirm. Correctly surfaced; no change needed to the plan.
