# Unit plan — fold the `ac` (climate) tile into `features/ac/`

Track C, Phase 3 / master-plan unit **F-devstate (ac slice)**. A single-tile fold
of the Climate · A/C tile (`tile_ac`) into a self-contained App at `features/ac/`,
following the just-landed **weather** multi-tile fold (commit `4be52f800`,
`features/weather/`) as the proven template, and the single-tile shape of
`features/network/` and `features/guest-wifi/`.

**Implementer:** a different agent. **Reviewer/verifier:** different agents. This
document is the whole spec; do not improvise beyond it without recording a
`PLACEHOLDER` decision.

---

## 0. Scope, non-goals, and the one hard constraint

**In scope (all in ONE atomic commit):**
- Move the climate tile + its full detail-modal web closure into `features/ac/web/`.
- Move the tRPC `climate` router into `features/ac/api.ts` (a `defineApi` facet).
- Move the climate read/write service into `features/ac/service.ts`, repointed off
  `apps/api` and onto `@www/core` (device-state store + HA client + `HaEntity` +
  `DeviceClimateState`) plus the feature's own `config.ts` / `deps.ts`.
- Move the climate service test into `features/ac/service.test.ts`.
- Add `features/ac/manifest.ts` with the tile placement copied **verbatim** from the
  deleted `tile-registry.ts` entry.
- Regenerate `features/_generated/*.gen.ts` via `bun run apps:gen`.
- Repoint every cross-importer (`server.ts` health probe, `detail/registry.ts`,
  `routers/index.ts`, `tile-registry.ts`) and delete the originals with **no shims**.

**Explicitly NOT in scope (hard constraint, honor exactly):**
- **The climate ENFORCER stays hand-wired and stays in `apps/api`.**
  `apps/api/src/services/climate-enforcer-service.ts` and its wiring in
  `apps/worker/src/index.ts` (via `apps/api/src/worker-deps.ts`) are **NOT touched**.
  The enforcer is a 1s Worker **interval** cycle, not a queue job (`jobs.ts`) and not
  a cron (`jobs.ts`/`defineCron`). Do **not** create a `features/ac/jobs.ts`. Do not
  move `runClimateEnforcerCycle`. The `worker → apps/api` import for the enforcer is
  the accepted roadmap interim (device-ownership hoist is a later unit; see roadmap
  `~/.claude/plans/merry-hugging-river.md` Phase 3 §deferred-follow-up and master plan
  lines 205–222).
- **No `features/ac/schema.ts`** and **no owned table.** Climate owns no Postgres
  table; it reads/writes the shared `device_state` row (`DeviceClimateState`, owned by
  `@www/core`). The generic "schema→schema.ts" fold hint does **not** apply here — this
  App has no schema facet. The `apps:check` dup-table validator is satisfied because the
  App declares no table.
- `controls-service.ts` (the AC-fan reader) stays in `apps/api`. It reads the climate
  `device_state` row directly via the store + `CLIMATE_DEVICE_ID`, never via
  `climate-service`, so the service move does not touch it (only the `CLIMATE_DEVICE_ID`
  source changes — see §5).

**The one hard constraint that dictates the whole backend shape:** a Biome
`noRestrictedImports` boundary rule bans `features/*` from importing `apps/api`. The
moved `service.ts` currently imports six things from `apps/api`
(`../config/identity`, `../db/device-state-store`, `../db/schema`, `../env`,
`../integrations/homeassistant`, `../integrations/homeassistant/types`). Every one must
be repointed to `@www/core` or to the feature's own `config.ts`/`deps.ts`. This is the
central work of the fold.

### 5.1 `CLIMATE_DEVICE_ID` — hoist to `@www/core` (recommended) — DECISION NEEDED

`CLIMATE_DEVICE_ID = "climate-thermostat"` is a cross-service **rendezvous ID**: it is
read by (a) the moving `service.ts`, (b) the staying `climate-enforcer-service.ts`,
(c) the staying `controls-service.ts`, plus ONE test (`climate-enforcer-service.test.ts`;
the service test `climate.test.ts` does NOT import it). Once `service.ts` lives in
`features/ac`, it cannot import it from `apps/api/src/config/identity.ts`. Re-declaring
the literal inside the feature would FORK a rendezvous ID across a package boundary —
exactly the anti-pattern the AGENTS.md "get IDs right up front / single source of truth"
invariant forbids.

**Recommended:** add `CLIMATE_DEVICE_ID` to `@www/core` (new
`packages/core/src/device-state/identity.ts`:
`export const CLIMATE_DEVICE_ID = "climate-thermostat";`) and re-export it from
`packages/core/src/index.ts` by adding the exact per-file barrel line
`export * from "./device-state/identity";` (the index uses per-file `export *`, not a
named list). Then:
- `features/ac/service.ts` imports it from `@www/core`.
- `apps/api/src/config/identity.ts` re-exports it: `export { CLIMATE_DEVICE_ID } from "@www/core";`
  (keeps `TOPOLOGY_ANCHOR_IP` / `DESK_RF_BONDED_UUID` local). This makes the change
  **zero-churn** for the two staying `apps/api` consumers (`climate-enforcer-service.ts`,
  `controls-service.ts`) + the one test (`climate-enforcer-service.test.ts`) — they keep
  importing from `../config/identity` unchanged. The enforcer file is NOT edited, honoring the
  "enforcer untouched" constraint (only its transitive constant source moves, and even
  its import line stays identical).

> **DECISION (resolved — hoist):** the `CLIMATE_DEVICE_ID` → `@www/core` hoist is the
> chosen approach for this unit; implement it as written above (touches shared
> `packages/core` + `index.ts`, plus the one-line re-export in
> `apps/api/src/config/identity.ts`). Review verified it is single-source-of-truth-correct
> and zero-churn for every staying importer. Do NOT use the feature-local re-declaration
> fallback.

### 5.3 `deps.ts` — the feature's own store + HA client

The feature builds its OWN instances from `@www/core` factories + its `config` slice
(the pattern `apps/api/src/integrations/homeassistant.ts` documents as the intended
end-state: "each caller builds its own instance from its config slice"). Both the
feature `service` and the `apps/api` enforcer then operate independent client/store
instances over the SAME HA + `device_state` row — correct, they are stateless adapters.

```ts
import { createHomeAssistantClient, createPgDeviceStateStore, deviceState } from "@www/core";
import { drizzle } from "drizzle-orm/node-postgres";
import { createPool } from "@www/core";
import { config } from "./config";

// The device_state store over this feature's own lazy pool. createPgDeviceStateStore
// only needs select/insert/update and references the core-owned `deviceState` table
// internally, so a bare drizzle handle (schema optional) suffices.
export const deviceStateStore = createPgDeviceStateStore(
  drizzle(createPool(config.DATABASE_URL), { schema: { deviceState } }),
);

// The env-free HA client bound to this feature's config slice.
export const ha = createHomeAssistantClient({ baseUrl: config.HA_URL, token: config.HA_TOKEN });
```

Verified: `createPgDeviceStateStore(db: PgDeviceStateDb)` needs only
`Pick<NodePgDatabase, "select"|"insert"|"update">` and imports `deviceState` itself, so
passing the `{ schema: { deviceState } }` is belt-and-braces, not required. The core HA
client exposes `isConfigured()`, `getEntities(domain)`, `callService(...)` — the exact
surface `service.ts` uses.
