## Why this unit exists

`media-path` is the pure photo-filename/path module (`node:fs/promises` + `node:path` only, no
db, no env). Today it lives at `apps/api/src/services/media-path.ts`. Track C folds the wake and
booth photo services into `features/*` (features `wakes` and `booth`), and `features/*` may not
import `apps/api/*` (Biome `noRestrictedImports`). So the shared filename/path primitive has to
sit in `packages/core` first. This unit only moves the module and repoints its consumers — the
wake/booth folds happen in later units.

## Key difference from the HA hoist (drives the shim decision)

The HA hoist kept thin re-export shims in `apps/api/src/integrations/homeassistant/` because HA
had (a) an **env-bound singleton** (`ha = createHomeAssistantClient({ baseUrl: env.HA_URL, ... })`)
that 23 consumers imported, and (b) that many consumers. `media-path` has **neither**: it exports
only pure functions, there is no env binding to preserve, and there are exactly **3** real
consumers, all inside `apps/api`. A re-export shim here would be an unused indirection that `bun
run knip` (zero-tolerance) flags as an unused export/file.

**Decision: repoint all consumers directly to `@www/core`. No shim.** Delete the old file
outright.

## Gotchas (honor these)

- **Shared main checkout.** Stage explicit paths (table above). NEVER `git add -A` / `git add .` —
  it swallows concurrent sessions' uncommitted work. Check `git show --stat HEAD` after commit.
- **`git pull --rebase --autostash` before push**, and again if push rejects. A peer rebase can
  unstage a `git rm`/rename — re-verify the old file is gone after any rebase.
- **knip zero-tolerance.** Because we chose NO shim, the old `media-path.ts` must be fully deleted
  (git mv handles this) and no barrel/index anywhere should re-export it from `apps/api`. If knip
  flags an unused export in `@www/core`, it means a consumer was missed — repoint it, don't delete
  the export.
- **`bun build` reads the CWD tsconfig, not the entry's** (see `bun-build-alias-needs-cwd-tsconfig`
  memory). Not exercised by this unit's verify chain (no direct `bun build` step), but if CI's
  api-image build goes red with an unresolved `@www/core`, that's the cause — `@www/core` is a
  normal workspace dep of apps/api already (HA hoist added it), so this should be a non-issue;
  confirm `apps/api/package.json` lists `@www/core` (the HA hoist's `bun.lock` change implies it
  does).
- **Foreground `gh run watch`.** Subagents die if they background-wait on CI (600s watchdog). Watch
  inline with `--exit-status`.
