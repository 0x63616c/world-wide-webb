## Findings

1. **[MAJOR] Task 4 Step 5's knip rationale is factually false, and the load-bearing `@public` tag is never mentioned.**
   The plan claims (Step 5, and Self-Review) "there should be none — every moved export has ≥1 consumer." This is wrong. Of the 6 exported types, **only `SonosFavorite` and `ZoneGroup` have consumers**; `MediaInfo`, `PositionInfo`, `TransportInfo`, and `ZoneMember` have **zero** consumers anywhere in the repo. knip stays green today *solely* because `apps/api/src/integrations/sonos/index.ts` carries a `/** @public … */` JSDoc tag above the `export type { … }` block, and `knip.jsonc` excludes `@public` exports via `"tags": ["-public"]`.
   The outcome is still safe — a content-identical `git mv` preserves the `@public` tag, and `tags` is a repo-global knip setting, so post-move knip passes exactly as it does now (same as the co-located media-path test precedent). But the plan's stated reason is misleading and would send an implementer down the wrong path if knip ever flagged these.
   **Fix:** (a) Replace the Step 5 / Self-Review claim with the truth: 4 of the 6 types are consumer-less and stay green via the load-bearing `@public` tag on `sonos/index.ts`. (b) Add an explicit gotcha to Task 1 Step 1: the `git mv` of `index.ts` must be byte-identical — the `@public` JSDoc block is load-bearing for knip; do not "tidy" or drop it.

2. **[MINOR] `DeviceStateStore` import line numbers are stated wrong for two services.**
   The plan (Task 2 Steps 2 & 4, and the ground-truth block) says both `sonos-volume-enforcer-service.ts` and `sonos-sound-system-service.ts` already import `DeviceStateStore` from `@www/core` "at :22". Actual: enforcer is **line 20**, sound-system is **line 23**. Non-load-bearing — the repoint edits target the `SonosClient` / `ZoneGroup` specifier lines (enforcer :25, sound-system :26–27), which are correct — but the stale line numbers should be corrected or dropped to avoid confusing the implementer.
   **Fix:** correct to :20 / :23, or omit the incidental line numbers.

3. **[MINOR / advisory] Barrel diverges from the just-landed HA precedent — worth a one-line acknowledgement.**
   The HA hoist (`da0be339e`, `packages/core/src/homeassistant/index.ts`) deliberately re-exports **only the names consumers depend on** (`HaEntity`, `createHomeAssistantClient`, `HaError`, `HomeAssistantClient`) and carries **no** `@public` tag — trimming the surface rather than annotating it. The sonos plan instead re-exports all 6 types and leans on `@public`. Both pass knip; the `@public` route is valid and preserves the client's existing public shape. No change required, but the plan's "mirror the HA/UniFi hoist" framing should note this one intentional divergence so a reviewer isn't surprised the barrel isn't trimmed to consumed names.
