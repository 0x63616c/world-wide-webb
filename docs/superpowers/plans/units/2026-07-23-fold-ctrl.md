# Unit F-ctrl — Fold the `tile_ctrl` (Controls) tile into `features/ctrl`

Track C, Wave 7 (device-state cluster). A **single-tile fold** in the same shape
the roadmap prescribes for `ac`: the tile's `web.tsx` / `api.ts` / `service.ts` /
`schema.ts` move into a self-contained `features/ctrl/` App that reads the shared
`@www/core` device-state store and HA client. The light **enforcer / party cycles
stay hand-wired** in `apps/api` + `apps/worker` (device-ownership logic deferred
per the master plan). ONE atomic backend+manifest commit.

Reference folds to mirror: `features/network/` (single-tile `tiles[]` manifest
shape post-F0), `features/weather/` (`config.ts` + `db.ts` + web-subtree move +
detail-wiring repoint precedent, commit `4be52f800`),
`features/guest-wifi/` (facet set). Pattern doc:
`docs/writing-scalable-typescript/README.md`.

> This unit is NOT as clean as `weather`. The controls **service** depends on
> three `apps/api/src/config/*` modules (`lights`, `identity`, `lamp-scenes`) that
> a folded feature cannot import (`features/* → apps/api` is Biome-banned). Two of
> them are shared substrate the roadmap parks in `@www/core`.
> **DECIDED (was PLACEHOLDER-1): option A — the `config/lights` +
> `config/identity` hoist into `@www/core` is folded INTO this same atomic commit.**
> `config/lamp-scenes` moves into `features/ctrl` (ctrl-domain). See **§THE CRUX**.

### Where each config module goes (the data-layout decision)

Resolved by **ownership**, honoring the 10x-100x "get data layout right up front"
invariant:

1. **`config/lamp-scenes.ts` → `features/ctrl/lamp-scenes.ts`** (ctrl-domain: lamp
   scenes, mood/party palettes, `LampMode`/`LampModeSpeed`, `partyColorsAtTick`).
   The two `apps/api` consumers that stay hand-wired — `light-enforcer-service.ts`
   and `party-service.ts` — repoint to `@features/ctrl/lamp-scenes`
   (`apps/api → @features` is the ALLOWED direction).

2. **`lamp_mode` table + `LAMP_MODE_SINGLETON_ID` → `features/ctrl/schema.ts`**
   (ctrl-owned singleton). `light-enforcer-service.ts` and `party-service.ts`
   repoint to `@features/ctrl/schema`. Standalone singleton, no FK / relation —
   safe to relocate; same SQL table name `lamp_mode`, no DDL change.

3. **`config/lights.ts` → `@www/core`** (the shared **device/light registry**:
   `LIGHTS`, `findLight`, `LAMP_ENTITY_IDS`, `FIXTURE_ENTITY_IDS`, `LightEntry`,
   `LightKind`, `LightControl`, `lightControl`, `Room`). This is NOT ctrl-owned —
   `light-enforcer-service.ts`, `party-service.ts`, `device-ownership.ts` and the
   controls service all read it. It is the registry half of the master plan's
   deferred "device-ownership hoist", and `@www/core` is its roadmap-final home.
   **Hoisted in THIS commit (option A).**

4. **`config/identity.ts` → `@www/core`** (`CLIMATE_DEVICE_ID`,
   `TOPOLOGY_ANCHOR_IP`, `DESK_RF_BONDED_UUID` — "cross-service device identity;
   no service owns another's identity"). Shared across climate/sonos/controls;
   belongs in core, not a tile feature. Pure consts, trivial hoist. **Hoisted in
   THIS commit (option A).**

> **Why NOT put `config/lights` / `config/identity` under `features/ctrl`:** that
> would make `device-ownership.ts`, `climate-service.ts`, `sonos-*` all depend on
> the ctrl *tile feature* for the shared device registry — cross-feature coupling
> through a tile, violating "shared primitives live in core". Rejected.

> **Why this does NOT contradict "device-ownership deferred":** the deferred piece
> is the ownership **logic** — `device-ownership.ts` (`ownerOf`/`DeviceOwner`) and
> every enforcer cycle **stay in `apps/api`/`apps/worker`, hand-wired**. Only the
> pure registry *data* (`config/lights`, `config/identity`) moves to core, where
> `device-ownership.ts` reads it via the allowed `apps/api → @www/core` direction.
> No enforcer moves; no worker rewiring of enforcer cycles.
